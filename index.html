Part 1 — Foundations
01
Spring Boot vs the Spring Framework

Same engine, but Boot removes the assembly required to start it.

Spring Framework gives you all the building blocks (DI, MVC, data access) but you wire everything yourself in XML or Java config. Spring Boot is Spring plus sane defaults — an embedded server, auto-configured beans, and one dependency ("starter") per feature — so you get a running app in minutes instead of days.
Amazon — checkout service

Before Boot, a team standing up Amazon's checkout service had to hand-configure a DispatcherServlet, deploy a WAR to a Tomcat server, wire a DataSource bean, and set up a view resolver — all before writing a single line of checkout logic.

With Spring Boot, the same team runs mvn spring-boot:run and gets an embedded Tomcat, JSON serialization, and a connection pool out of the box. They spend day one writing the CheckoutController, not XML.

// This ONE class is a complete, runnable web application in Spring Boot.
// No web.xml, no external Tomcat install, no manually registered servlets.

@SpringBootApplication   // = @Configuration + @EnableAutoConfiguration + @ComponentScan
public class AmazonCheckoutApplication {
    public static void main(String[] args) {
        SpringApplication.run(AmazonCheckoutApplication.class, args);
        // Embedded Tomcat starts on :8080 automatically.
    }
}
💡
TipSpring Boot is not a replacement for Spring — every @Service, @Autowired, and ApplicationContext you use is still plain Spring. Boot only automates the setup.

IoC Container & Dependency Injection

Classes declare what they need; Spring hands them a ready-made instance.

Instead of a class creating its own dependencies with new, it just declares what it needs in its constructor. Spring's IoC (Inversion of Control) container builds every bean once at startup and injects the right instances wherever they're needed. You get loose coupling and classes that are trivial to test.
Amazon — placing an order

OrderService needs to charge a card and reduce stock. It doesn't know or care how PaymentService talks to the payment gateway, or how InventoryService talks to the warehouse database — it just asks for both in its constructor, and Spring supplies them.

@Service
public class PaymentService {
    public void charge(String orderId, BigDecimal amount) {
        // talks to the payment gateway
    }
}

@Service
public class InventoryService {
    public void reserveStock(String sku, int qty) {
        // decrements warehouse stock
    }
}

@Service
public class OrderService {

    private final PaymentService paymentService;
    private final InventoryService inventoryService;

    // Constructor injection: Spring sees this constructor, finds a
    // PaymentService bean and an InventoryService bean, and passes them in.
    // No "new PaymentService()" anywhere in this class.
    public OrderService(PaymentService paymentService,
                         InventoryService inventoryService) {
        this.paymentService = paymentService;
        this.inventoryService = inventoryService;
    }

    public void placeOrder(String orderId, String sku, int qty, BigDecimal amount) {
        inventoryService.reserveStock(sku, qty);
        paymentService.charge(orderId, amount);
    }
}
⚠️
PitfallField injection (@Autowired directly on a field) works but hides a class's real dependencies and makes it hard to unit test without Spring. Constructor injection — as above — makes dependencies explicit and lets you build the class in a plain JUnit test with new OrderService(mockPayment, mockInventory).

Auto-Configuration & Starters

Adding one dependency wires up an entire feature for you.

A "starter" is a bundle of dependencies for one feature (data access, security, web). Auto-configuration is Spring Boot inspecting what's on your classpath at startup and registering the beans that feature needs — only if you haven't already configured them yourself.
Instagram — adding database support

A backend engineer wants Instagram's post service to save posts to MySQL. They add one line to pom.xml: spring-boot-starter-data-jpa. On startup, Boot detects a JPA implementation (Hibernate) and a JDBC driver on the classpath, and auto-configures a DataSource, an EntityManagerFactory, and a PlatformTransactionManager — nobody wrote that configuration by hand.

<!-- pom.xml — this single starter pulls in Hibernate, HikariCP connection
     pool, and Spring's JPA support, and triggers their auto-configuration -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
// A simplified version of what Spring Boot auto-configures behind the
// scenes once it sees spring-boot-starter-data-jpa + a MySQL driver.
// You never write this class yourself — @ConditionalOnClass triggers it.

@Configuration
@ConditionalOnClass(EntityManagerFactory.class)     // only if JPA is on the classpath
@ConditionalOnMissingBean(DataSource.class)          // only if YOU haven't defined one
public class InstagramJpaAutoConfiguration {

    @Bean
    public DataSource dataSource(DataSourceProperties props) {
        return props.initializeDataSourceBuilder().build();
    }
}
💡
TipYou can always see exactly what auto-configured and why: run with --debug and check the "CONDITIONS EVALUATION REPORT," or hit /actuator/conditions if Actuator is on the classpath.

Stereotype Annotations

@Component, @Service, @Repository, @Controller — labels that tell Spring what a class does.

All four are technically the same thing — they mark a class as a Spring-managed bean so component scanning picks it up. But each name signals intent and unlocks small extra behavior: @Repository auto-translates database exceptions, @Controller plugs into Spring MVC, @Service signals business logic, @Component is the generic fallback.
YouTube — the upload path

YouTube's "upload a video" flow has one class per layer, each labeled by role: a controller receives the HTTP upload, a service applies business rules (max length, encoding), and a repository saves metadata.

@RestController                                 // handles HTTP requests, returns JSON
@RequestMapping("/api/videos")
public class VideoController {

    private final VideoService videoService;

    public VideoController(VideoService videoService) {
        this.videoService = videoService;
    }

    @PostMapping
    public ResponseEntity<VideoResponse> upload(@RequestBody UploadRequest request) {
        Video saved = videoService.upload(request);
        return ResponseEntity.ok(VideoResponse.from(saved));
    }
}

@Service                                        // business logic: rules, orchestration
public class VideoService {

    private final VideoRepository videoRepository;

    public VideoService(VideoRepository videoRepository) {
        this.videoRepository = videoRepository;
    }

    public Video upload(UploadRequest request) {
        if (request.durationSeconds() > 43200) {
            throw new IllegalArgumentException("Video exceeds 12-hour limit");
        }
        Video video = new Video(request.title(), request.durationSeconds());
        return videoRepository.save(video);
    }
}

@Repository                                     // data access — DB exceptions become
public interface VideoRepository                //   Spring's DataAccessException, not
        extends JpaRepository<Video, Long> {}   //   a raw driver-specific one
💡
Tip@RestController is just @Controller + @ResponseBody combined — every method's return value is written straight to the HTTP response body as JSON instead of being resolved to a view name.

Spring MVC — REST Controllers

Annotations map HTTP requests straight to Java methods.

Spring MVC's DispatcherServlet receives every incoming HTTP request and routes it to the right method based on annotations like @GetMapping and @PostMapping. Method parameters are extracted from the URL, query string, or JSON body automatically — you write plain Java and get a REST API.
Amazon — product catalog API

The product page needs to fetch one product by ID, and the search page needs to list products by category with pagination. Two endpoints, two annotated methods.

@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    // GET /api/products/B08N5WRWNW  -> {id} is pulled from the URL path
    @GetMapping("/{id}")
    public ResponseEntity<ProductResponse> getProduct(@PathVariable String id) {
        return productService.findById(id)
                .map(p -> ResponseEntity.ok(ProductResponse.from(p)))
                .orElse(ResponseEntity.notFound().build());
    }

    // GET /api/products?category=electronics&page=0&size=20
    @GetMapping
    public Page<ProductResponse> listProducts(
            @RequestParam String category,
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        return productService.findByCategory(category, PageRequest.of(page, size))
                .map(ProductResponse::from);
    }

    // POST /api/products  with a JSON body -> deserialized into NewProductRequest
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public ProductResponse createProduct(@RequestBody NewProductRequest request) {
        return ProductResponse.from(productService.create(request));
    }
}
💡
TipResponseEntity lets you control the exact HTTP status and headers (404 for a missing product), while a plain return type like Page<ProductResponse> always returns 200. Use ResponseEntity whenever the outcome can genuinely differ.

Spring Data JPA

Write an interface; Spring generates the SQL-executing implementation.

You define a repository interface extending JpaRepository, and Spring generates a working implementation at startup — no DAO boilerplate, no hand-written SQL for common cases. Method names like findByUserIdOrderByCreatedAtDesc are parsed and turned into queries automatically.
Instagram — loading a user's feed

The feed screen needs the newest posts from a given user, and separately, a count of how many posts they've made this week for a "trending creator" badge. One derived query method, one JPQL query — both without a single line of raw SQL.

public interface PostRepository extends JpaRepository<Post, Long> {

    // Derived query: Spring parses the method name and builds the SQL for you.
    // "findBy" + "UserId" + "OrderByCreatedAtDesc" ->
    //   SELECT * FROM post WHERE user_id = ? ORDER BY created_at DESC
    List<Post> findByUserIdOrderByCreatedAtDesc(Long userId, Pageable pageable);

    // JPQL for anything a method name can't express cleanly
    @Query("SELECT COUNT(p) FROM Post p WHERE p.userId = :userId AND p.createdAt >= :since")
    long countRecentPosts(@Param("userId") Long userId, @Param("since") Instant since);
}

@Service
public class FeedService {

    private final PostRepository postRepository;

    public FeedService(PostRepository postRepository) {
        this.postRepository = postRepository;
    }

    public List<Post> loadFeed(Long userId, int page) {
        return postRepository.findByUserIdOrderByCreatedAtDesc(
                userId, PageRequest.of(page, 10));
    }

    public boolean isTrendingThisWeek(Long userId) {
        Instant weekAgo = Instant.now().minus(7, ChronoUnit.DAYS);
        return postRepository.countRecentPosts(userId, weekAgo) >= 5;
    }
}
⚠️
PitfallLoading a feed of posts and then accessing post.getComments() in a loop, one post at a time, triggers a separate SQL query per post — the classic N+1 problem. Fix it with a JPQL JOIN FETCH or an @EntityGraph so comments load in the same query as the posts.

Bean Scopes & Lifecycle

How many instances Spring creates, and the hooks around their birth and death.

By default every Spring bean is a singleton — one shared instance for the whole app. Prototype scope creates a new instance every time it's requested. @PostConstruct runs right after a bean's dependencies are injected; @PreDestroy runs right before the app shuts down — useful for opening and closing expensive resources exactly once.
WhatsApp — the connection manager

WhatsApp's backend keeps one pool of persistent connections to its message broker for the entire application — creating a new pool per request would exhaust resources instantly. That's a singleton by nature, and it needs to open the pool on startup and close it cleanly on shutdown so no messages are dropped mid-flight.

@Component   // singleton by default — Spring creates exactly ONE instance
public class MessageBrokerConnectionManager {

    private BrokerConnectionPool pool;

    @PostConstruct   // runs once, right after this bean's fields/dependencies are set
    public void openConnections() {
        pool = BrokerConnectionPool.open(50);   // 50 persistent connections
        System.out.println("Broker pool opened with 50 connections");
    }

    public void send(Message message) {
        pool.borrowConnection().send(message);
    }

    @PreDestroy   // runs once, right before the application context shuts down
    public void closeConnections() {
        pool.drainAndClose();   // finish in-flight sends before closing
        System.out.println("Broker pool drained and closed");
    }
}

// Contrast: a PDF invoice generator used per-request needs fresh, isolated state
// (a mutable document builder) each time it's injected — prototype scope fits here.
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class InvoicePdfBuilder {
    private final StringBuilder content = new StringBuilder();  // must not be shared
}
⚠️
PitfallInjecting a prototype bean into a singleton only creates it once — at the singleton's construction time — because the singleton holds onto that single instance forever after. To get a fresh prototype on every call, inject an ObjectProvider<InvoicePdfBuilder> and call .getObject() each time.

Configuration Properties & Profiles

Same code, different settings for dev, staging, and production.

Hard-coding URLs, keys, and limits means rebuilding the app for every environment. Spring Boot externalizes them into application.yml files, with @ConfigurationProperties binding a whole block of settings to a typed Java object, and profiles letting you swap entire config blocks based on which environment is active.
WhatsApp — SMS gateway for OTP verification

WhatsApp's phone-verification service sends OTPs through a sandbox SMS gateway in development and a real, rate-limited provider in production — same code, different endpoint, key, and retry limit per environment.

---

sms-gateway:
  base-url: https://sandbox.smsgateway.test

  base-url: https://api.smsgateway.com
  api-key: ${SMS_GATEWAY_KEY}   # injected from an environment variable, never committed
  retry-attempts: 5

Validation

Reject bad input before it ever reaches your business logic.

Bean Validation annotations (@NotBlank, @Email, @Size) go directly on a request object's fields. Adding @Valid in the controller tells Spring to check every rule before the method body runs — invalid input never reaches your service layer at all.
Instagram — sign-up form

A new account needs a real-looking email, a username of reasonable length, and a password meeting a minimum strength — checked in one place, consistently, instead of scattered if statements across the codebase.

public record SignupRequest(

    @NotBlank(message = "Username is required")
    @Size(min = 3, max = 30, message = "Username must be 3-30 characters")
    String username,

    @NotBlank(message = "Email is required")
    @Email(message = "Email must be a valid address")
    String email,

    @NotBlank(message = "Password is required")
    @Size(min = 8, message = "Password must be at least 8 characters")
    String password
) {}

@RestController
@RequestMapping("/api/auth")
public class AuthController {

    @PostMapping("/signup")
    public ResponseEntity<Void> signup(@Valid @RequestBody SignupRequest request) {
        // If validation fails, this method body never runs —
        // Spring throws MethodArgumentNotValidException first.
        accountService.createAccount(request);
        return ResponseEntity.status(HttpStatus.CREATED).build();
    }
}
💡
TipPair this with a @ControllerAdvice (next chapter) that catches MethodArgumentNotValidException and turns it into a clean JSON error listing every failed field — instead of Spring's default, noisy stack-trace response.

Global Exception Handling

One place to turn every exception into a clean, consistent API response.

Without central handling, every controller ends up with its own scattered try/catch blocks, and clients get inconsistent error shapes. @RestControllerAdvice plus @ExceptionHandler intercepts exceptions thrown anywhere in any controller and maps them to one consistent JSON error format and HTTP status.
Amazon — checkout failure

A shopper tries to buy the last unit of a product that just sold out to someone else a second earlier. InventoryService throws OutOfStockException — and every endpoint in the app that can run out of stock returns the exact same clean 409 response, without repeating the same catch block everywhere.

public class OutOfStockException extends RuntimeException {
    public OutOfStockException(String sku) {
        super("Product " + sku + " is out of stock");
    }
}

public record ApiError(String message, int status) {}

@RestControllerAdvice   // applies to every @RestController in the app
public class GlobalExceptionHandler {

    @ExceptionHandler(OutOfStockException.class)
    public ResponseEntity<ApiError> handleOutOfStock(OutOfStockException ex) {
        ApiError error = new ApiError(ex.getMessage(), 409);
        return ResponseEntity.status(HttpStatus.CONFLICT).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiError> handleValidation(MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult().getFieldErrors().stream()
                .map(FieldError::getDefaultMessage)
                .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest().body(new ApiError(message, 400));
    }

    @ExceptionHandler(Exception.class)   // catch-all safety net
    public ResponseEntity<ApiError> handleUnexpected(Exception ex) {
        return ResponseEntity.internalServerError()
                .body(new ApiError("Something went wrong", 500));
    }
}
⚠️
PitfallOrder matters: put specific exception handlers (OutOfStockException) before the generic Exception.class catch-all. Spring picks the most specific match, but a sloppy catch-all higher up can accidentally swallow errors you meant to handle precisely.

AOP — Aspect-Oriented Programming

Cross-cutting concerns like logging and auditing, without touching business code.

Some concerns — logging, auditing, timing, retries — apply across many unrelated methods. Instead of copy-pasting logging code into every method, AOP lets you write it once as an "aspect" and declare, with a @Pointcut expression, which methods it should wrap around.
Amazon — audit trail on checkout

Every method in the checkout package needs to be logged for compliance: who called it, with what arguments, how long it took. Writing that logging code inside placeOrder, applyDiscount, and every other checkout method would be repetitive and easy to forget on new methods.

@Aspect
@Component
public class CheckoutAuditAspect {

    private static final Logger log = LoggerFactory.getLogger(CheckoutAuditAspect.class);

    // Matches every public method in every class under the checkout package —
    // new methods added later are covered automatically, no extra wiring needed.
    @Around("execution(public * com.amazon.checkout..*.*(..))")
    public Object auditCall(ProceedingJoinPoint joinPoint) throws Throwable {
        String method = joinPoint.getSignature().toShortString();
        long start = System.currentTimeMillis();

        log.info("AUDIT: entering {} with args={}", method, joinPoint.getArgs());
        Object result = joinPoint.proceed();   // actually runs placeOrder(), etc.
        long durationMs = System.currentTimeMillis() - start;

        log.info("AUDIT: {} completed in {}ms", method, durationMs);
        return result;
    }
}
💡
Tip@Around is the most powerful advice type because it wraps the method entirely — you decide whether joinPoint.proceed() even runs. For simpler cases, @Before and @AfterReturning are easier to reason about and harder to misuse.

Transactions — @Transactional

Multiple database writes that must all succeed together, or none at all.

@Transactional wraps a method in a database transaction: if every statement inside succeeds, the changes commit together; if any exception is thrown, everything rolls back as if none of it happened. It's the difference between "partially broken data" and "nothing changed."
Amazon — placing an order

Placing an order does three things: reserve stock, charge the card, create the order record. If the payment gateway fails after stock was already reserved, that reservation must be undone — otherwise the item is stuck "reserved" forever with no successful order behind it.

@Service
public class OrderService {

    private final InventoryRepository inventoryRepository;
    private final PaymentService paymentService;
    private final OrderRepository orderRepository;

    public OrderService(InventoryRepository inventoryRepository,
                         PaymentService paymentService,
                         OrderRepository orderRepository) {
        this.inventoryRepository = inventoryRepository;
        this.paymentService = paymentService;
        this.orderRepository = orderRepository;
    }

    @Transactional   // all three steps commit together, or none do
    public Order placeOrder(String sku, int qty, BigDecimal amount) {
        inventoryRepository.reserveStock(sku, qty);       // step 1: DB write

        boolean charged = paymentService.charge(amount);   // step 2: external call
        if (!charged) {
            throw new PaymentFailedException("Card declined");
            // exception thrown -> Spring rolls back step 1 automatically
        }

        return orderRepository.save(new Order(sku, qty, amount));  // step 3: DB write
    }
}
⚠️
PitfallCalling an @Transactional method from another method in the same class bypasses the transaction entirely — Spring implements it via a proxy that wraps external calls only, and self-invocation skips the proxy. Call it through an injected bean (a different class) to get the transaction to actually apply.

Caching — @Cacheable

Skip the database entirely for data that doesn't change every second.

@Cacheable stores a method's return value the first time it runs for a given set of arguments. Every later call with the same arguments returns the cached value instantly, skipping the method body — and the database — completely, until the cache entry is evicted or expires.
YouTube — video metadata on a viral video

A video's title, thumbnail, and channel name barely change, but a viral video can be requested millions of times an hour. Hitting the database for the same unchanged metadata on every single view would overwhelm it for no reason.

@Service
public class VideoMetadataService {

    private final VideoRepository videoRepository;

    public VideoMetadataService(VideoRepository videoRepository) {
        this.videoRepository = videoRepository;
    }

    // First call for a given videoId hits the DB and stores the result under
    // the cache name "videoMetadata" keyed by videoId. Every call after that,
    // for that same videoId, returns the cached object -- this method body
    // does not run again until the entry is evicted.
    @Cacheable(value = "videoMetadata", key = "#videoId")
    public VideoMetadata getMetadata(String videoId) {
        return videoRepository.findMetadataById(videoId);
    }

    // Called whenever a creator edits their video's title/thumbnail --
    // removes the stale entry so the next read re-fetches fresh data.
    @CacheEvict(value = "videoMetadata", key = "#videoId")
    public void updateTitle(String videoId, String newTitle) {
        videoRepository.updateTitle(videoId, newTitle);
    }
}
💡
TipThe default ConcurrentMapCacheManager is in-memory and per-instance — fine for a demo, useless across multiple app servers. For a real multi-instance deployment, back @Cacheable with Redis so every instance shares the same cache.

Async Processing — @Async

Let the slow part run in the background instead of making the user wait.

@Async makes a method run on a separate thread pool instead of the caller's thread. The caller moves on immediately without waiting for that method to finish — perfect for work the user doesn't need to wait for, like sending notifications.
Instagram — notifying followers of a new post

When someone with 50,000 followers posts a photo, notifying every follower can't block the "Post" button response — the poster should see "Posted!" in under a second, while notifications go out in the background over the next few seconds.

@SpringBootApplication
@EnableAsync   // turns on @Async processing for the whole application
public class InstagramApplication { }

@Service
public class PostService {

    private final PostRepository postRepository;
    private final NotificationService notificationService;

    public PostService(PostRepository postRepository,
                        NotificationService notificationService) {
        this.postRepository = postRepository;
        this.notificationService = notificationService;
    }

    public Post createPost(NewPostRequest request) {
        Post saved = postRepository.save(new Post(request));

        // Returns immediately -- doesn't wait for 50,000 notifications to send.
        notificationService.notifyFollowers(saved);

        return saved;   // the "Posted!" response goes back to the user right now
    }
}

@Service
public class NotificationService {

    @Async   // runs on a background thread pool, not the web request thread
    public void notifyFollowers(Post post) {
        List<Long> followerIds = followerRepository.findFollowerIds(post.getUserId());
        for (Long followerId : followerIds) {
            pushNotificationClient.send(followerId, post.getUserName() + " just posted");
        }
    }
}
⚠️

Scheduling — @Scheduled

Run a job automatically on a timer, without an external cron server.

@Scheduled runs a method automatically at a fixed rate, a fixed delay, or on a cron expression — no separate cron daemon, no external scheduler service needed for simple periodic jobs running inside your own app.
Amazon — abandoned cart reminder

Every night at 2 AM, Amazon needs to find carts that have sat untouched for 24 hours and email a "you left something in your cart" reminder — a routine job that just needs to run reliably on a schedule.

@SpringBootApplication
@EnableScheduling   // turns on @Scheduled processing for the application
public class AmazonCartApplication { }

@Component
public class AbandonedCartJob {

    private final CartRepository cartRepository;
    private final EmailService emailService;

    public AbandonedCartJob(CartRepository cartRepository, EmailService emailService) {
        this.cartRepository = cartRepository;
        this.emailService = emailService;
    }

    // cron: second minute hour day-of-month month day-of-week
    // Runs every night at 2:00 AM server time.
    @Scheduled(cron = "0 0 2 * * *")
    public void sendAbandonedCartReminders() {
        Instant cutoff = Instant.now().minus(24, ChronoUnit.HOURS);
        List<Cart> abandoned = cartRepository.findUntouchedSince(cutoff);

        for (Cart cart : abandoned) {
            emailService.sendReminder(cart.getUserEmail(), cart.getItems());
        }
    }
}

Spring Security

A chain of filters decides who's allowed to do what, before your code ever runs.

Every request passes through a chain of security filters first — checking authentication (who are you?) and then authorization (are you allowed to do this?). @PreAuthorize lets you write that authorization rule as an expression right on the method it protects.
Instagram — private account posts

A private account's posts should be visible to the owner and their accepted followers only — a stranger or a pending (not-yet-accepted) follower gets a 403, not the photo.

@Service
public class ProfileAccessService {

    private final FollowRepository followRepository;

    public ProfileAccessService(FollowRepository followRepository) {
        this.followRepository = followRepository;
    }

    public boolean canView(Long viewerId, Long profileOwnerId, AccountType accountType) {
        if (viewerId.equals(profileOwnerId)) return true;         // it's your own profile
        if (accountType == AccountType.PUBLIC) return true;        // anyone can view public

        return followRepository.existsByFollowerIdAndFollowingIdAndStatus(
                viewerId, profileOwnerId, FollowStatus.ACCEPTED);   // must be an accepted follower
    }
}

@RestController
@RequestMapping("/api/users/{userId}/posts")
public class ProfilePostsController {

    private final ProfileAccessService accessService;
    private final PostService postService;

    public ProfilePostsController(ProfileAccessService accessService, PostService postService) {
        this.accessService = accessService;
        this.postService = postService;
    }

    // #userId refers to the {userId} path variable; principal is the logged-in user
    @PreAuthorize("@profileAccessService.canView(principal.id, #userId, @postService.accountTypeOf(#userId))")
    @GetMapping
    public List<PostResponse> getPosts(@PathVariable Long userId) {
        return postService.getPostsFor(userId);   // only reached if canView() returned true
    }
}
💡
Tip@PreAuthorize runs before the method body — a denied check throws AccessDeniedException and the controller code never executes at all, which is exactly what you want for sensitive data like private posts.

NoSQL — Spring Data MongoDB

A document store for data whose shape changes often.

Relational databases force every record into the same fixed columns — great for orders and payments, painful when every record looks different. MongoDB stores JSON-like documents instead: each document can carry its own fields, and adding a new field never requires a schema migration. Spring Data MongoDB gives you the exact same repository pattern you already know from JPA (Chapter 6) — same derived query methods, same save()/findById() — just backed by collections of documents instead of tables of rows. Reach for it when your data is naturally nested or its shape keeps evolving.
Instagram — Stories

Think about what a Story actually contains: one might be a plain photo, the next a video with a poll sticker and a location tag, the next a text card with a music clip and a countdown timer. Every new sticker type Instagram ships would mean another nullable column in a SQL table — after two years you'd have a table with 40 columns where most are empty on any given row.

As a document, each Story simply carries whatever it has and nothing more. When the team ships a new "Add Yours" sticker, old Stories don't change, new Stories just include the new field, and no database migration runs at 2 AM.

@Document(collection = "stories")   // a collection is MongoDB's version of a table

    private Long userId;
    private String mediaUrl;
    private Instant createdAt;

    // stores only the stickers it actually has. No schema change to add new types.
    private Map<String, Object> stickers;
}

// Same repository pattern as JPA -- Spring generates the implementation,
// and the derived query method syntax is identical.
public interface StoryRepository extends MongoRepository<Story, String> {
    List<Story> findByUserIdOrderByCreatedAtDesc(Long userId);
    long countByUserIdAndCreatedAtAfter(Long userId, Instant since);
}
💡

Redis as a Primary Data Store

Not just a cache — a fast key-value store with built-in expiry.

Chapter 13 used Redis behind @Cacheable, where the database is still the source of truth and Redis just holds copies. But Redis also works as the actual home for a category of data: short-lived, high-churn values like sessions, OTP codes, presence status, and rate-limit counters. This data has two defining traits — it changes constantly, and it's worthless after a short time — which is exactly what Redis's in-memory speed and native TTL (time-to-live) expiry are built for. Writing it to MySQL would mean constant writes to disk for data you'll delete in seconds anyway.
WhatsApp — online presence

Open a chat and you see "online" or "last seen at 4:32 PM." Behind that, every connected phone sends a heartbeat every few seconds. With hundreds of millions of users doing this simultaneously, storing presence in a SQL table would mean an enormous, pointless write load — and stale rows whenever a phone loses signal in a tunnel without saying goodbye.

The Redis version is elegant: each heartbeat sets a key with a 30-second TTL. As long as heartbeats keep arriving, the key keeps getting refreshed and the user shows "online." The moment heartbeats stop — battery died, network dropped, app killed — the key silently expires on its own, and the user flips to "last seen." No cleanup job, no stale state, ever.

@Service
public class PresenceService {

    private final StringRedisTemplate redisTemplate;

    public PresenceService(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    // Called on every heartbeat from the phone (every ~10 seconds).
    public void markOnline(String userId) {
        redisTemplate.opsForValue()
                .set("presence:" + userId, "online", Duration.ofSeconds(30));
        // TTL is the whole trick: if heartbeats stop, the key deletes itself.
        // Disconnection handling requires zero code.
    }

    public boolean isOnline(String userId) {
        return redisTemplate.hasKey("presence:" + userId);
    }

    // Bonus: an atomic counter, another classic "Redis as a store" use case --
    // e.g. counting failed login attempts before locking an account.
    public long recordFailedLogin(String userId) {
        String key = "failed-logins:" + userId;
        Long attempts = redisTemplate.opsForValue().increment(key);
        redisTemplate.expire(key, Duration.ofMinutes(15));
        return attempts;
    }
}
⚠️
PitfallRedis is in-memory: a restart without persistence configured wipes everything. That's fine for presence and OTPs (they'd expire anyway) — but never let "we already have Redis" tempt you into storing data you can't afford to lose, like orders, without understanding Redis persistence (RDB/AOF) first.

Wide-Column NoSQL — Cassandra

Built for massive write throughput across many servers.

A single MySQL server eventually hits a ceiling on how many writes per second it can absorb — and scaling relational writes across machines is genuinely hard. Cassandra was designed the other way around: it's a cluster of equal nodes with no single leader, where data is spread across nodes by a partition key, so writes scale almost linearly as you add machines. The trade-off is real: you give up joins, flexible ad-hoc queries, and strong consistency by default. You design each table around exactly the query you'll run against it — query-first design — rather than normalizing data and querying it however you like later.
YouTube — view counters

A viral video receives tens of thousands of "count this view" writes per second, from every continent at once. Against a single relational row, those writes would queue up behind one row lock — the hotter the video, the worse the contention, precisely when you can least afford it.

In Cassandra, the video ID is the partition key, so each video's writes land on a specific set of nodes, and different viral videos hit different nodes — the load spreads across the cluster instead of piling onto one row. This is the same reason Cassandra-family databases sit behind Netflix's viewing history and Discord's message storage: brutal, sustained write volume.

@Table("view_counts")
public class ViewCount {

    // The partition key decides WHICH NODES own this row.
    // All writes for one video go to the same partition -- and different
    // videos map to different nodes, spreading the total load.
    @PrimaryKey
    private String videoId;

    private long count;
}

public interface ViewCountRepository extends CassandraRepository<ViewCount, String> {

    // Queries MUST go through the partition key. This works:
    Optional<ViewCount> findByVideoId(String videoId);

    // But "find all videos with count > 1M" (no partition key) would need a
    // full cluster scan -- Cassandra rejects it by default. If you need that
    // query, you design a SECOND table shaped around it. Query-first design.
}
💡
TipThe interview-ready one-liner: choose Cassandra when writes dominate and you know your queries up front; choose a relational database when you need flexible queries, joins, and transactions. If someone asks "why not just MySQL?" — the answer is write scalability without a single point of contention.

Search Engines — Elasticsearch

Full-text, typo-tolerant search that a SQL "LIKE" can't match.

SQL's LIKE '%headphones%' does substring matching: it can't rank results by relevance, can't forgive a typo, can't understand that "running shoes" and "shoe for running" mean the same thing — and on a large table it's slow, because it can't use a normal index. Elasticsearch solves this with an inverted index: at write time, text is broken into analyzed terms, and each term maps to the documents containing it — like the index at the back of a book. Searches become fast term lookups with relevance scoring, fuzzy matching for typos, and stemming built in. The standard architecture: your primary database stays the source of truth, and you sync searchable fields into Elasticsearch.
Amazon — product search

A customer types "wireles headphnes" — two typos — into the search bar. They still expect to see "Sony Wireless Headphones" at the top, ranked by relevance and popularity, with results appearing as fast as they can type. A LIKE query returns nothing (the typo breaks the substring match), and even without typos it can't decide which of 4,000 matching products deserves the first slot.

Elasticsearch's analyzed index matches despite the typos (fuzzy matching), scores each product by how well it matches, and returns the ranked page in milliseconds — across a catalog of hundreds of millions of items. This is also what powers search at scale in apps like LinkedIn, Uber Eats, and Netflix's title search.

// This document lives in an Elasticsearch INDEX -- typically synced from the
// primary product database (via events or a pipeline), not written directly.
@Document(indexName = "products")
public class ProductDoc {
    @Id private String id;

    @Field(type = FieldType.Text)      // Text = analyzed for full-text search
    private String name;

    @Field(type = FieldType.Text)
    private String description;

    @Field(type = FieldType.Double)    // used for sorting/boosting, not full-text
    private double rating;
}

public interface ProductSearchRepository extends ElasticsearchRepository<ProductDoc, String> {
    // Full-text search across name and description -- analyzed, ranked, fast.
    List<ProductDoc> findByNameContainingOrDescriptionContaining(String name, String description);
}
⚠️
PitfallElasticsearch is not a primary database — it's eventually consistent, and losing an index should always be recoverable by re-syncing from the source of truth. Keep products in MySQL/PostgreSQL, mirror the searchable fields into Elasticsearch, and treat the index as rebuildable.

Choosing a Cache Provider

Caffeine, Redis, and Ehcache solve the same problem at different scales.

The @Cacheable annotation from Chapter 13 is provider-agnostic — what actually stores the cached values is the CacheManager bean, and that's where the real decision lives. Caffeine keeps entries in the JVM heap: nanosecond reads, zero network hops, but each app instance has its own private copy. Redis keeps entries in a shared external server: one cache for all instances, survives app restarts, at the cost of a ~1ms network round trip. Ehcache sits between, adding disk overflow to in-memory caching. The decision rule: one instance → Caffeine; multiple instances that must see the same cached data → Redis; many teams run both as two tiers (check Caffeine first, then Redis, then the database).
YouTube — same cache, wrong provider

Imagine video metadata cached with Caffeine across 20 server instances. A creator fixes a typo in their title — the instance that handled the edit evicts its local entry, but the other 19 instances keep serving the stale title from their own private caches for the rest of the TTL. Viewers see different titles depending on which server they hit.

Switch the CacheManager to Redis and there's exactly one cached copy shared by all 20 instances: one eviction, everyone consistent. The service code with @Cacheable doesn't change by a single character.

// Option A: Caffeine -- blazing fast, but private to each instance.
@Bean
public CacheManager caffeineCacheManager() {
    CaffeineCacheManager manager = new CaffeineCacheManager();
    manager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(10_000));          // bound it -- unbounded caches eat the heap
    return manager;
}

// Option B: Redis -- one shared cache for every instance. Same @Cacheable code.
@Bean
public CacheManager redisCacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10));
    return RedisCacheManager.builder(factory).cacheDefaults(config).build();
}
💡
TipAlways set both a TTL and a maximum size on in-memory caches. An unbounded Caffeine cache is a slow-motion memory leak — it grows until the JVM starts long GC pauses, and the "cache" becomes the reason the service is slow.

Circuit Breaker — Resilience4j

Stop hammering a dependency that's already failing.

When a dependency starts failing, the naive behavior — keep calling it — makes everything worse: your threads pile up waiting on timeouts, your users wait, and the struggling dependency gets buried under even more load it can't handle. A circuit breaker works like the electrical one in your house. It watches the failure rate through a sliding window of recent calls. Below the threshold, the circuit is CLOSED and calls flow normally. Cross the threshold and it OPENS: calls fail instantly (or route to a fallback) without touching the dependency at all, giving it room to recover. After a cool-down it goes HALF-OPEN, letting a few trial calls through — success closes the circuit, failure re-opens it.
Amazon — third-party payment gateway degrades

The external payment gateway starts timing out at 10 seconds per call during a traffic spike on their side. Without a breaker, every checkout thread on every Amazon server spends 10 seconds stuck waiting — the thread pool drains, and soon even browsing (which doesn't touch payments at all) is slow, because there are no threads left to serve it. One sick dependency has taken down the whole storefront. This exact failure mode — cascading failure — is why Netflix built Hystrix, the ancestor of Resilience4j.

With the breaker: after the failure rate crosses 50%, the circuit opens. Checkouts instantly get the fallback — "order accepted, payment processing, we'll email you" — the order is queued for retry, threads stay free, and the gateway gets breathing room to recover. Sixty seconds later, the half-open trial calls find it healthy and normal payments resume, all without anyone paging the on-call engineer.

@Service
public class PaymentGatewayClient {

    @CircuitBreaker(name = "paymentGateway", fallbackMethod = "paymentFallback")
    public PaymentResult charge(BigDecimal amount) {
        return restClient.post().uri("https://gateway/charge")
                .body(amount).retrieve().body(PaymentResult.class);
    }

    // Runs when the circuit is OPEN (no call attempted) or the call throws.
    // Signature: same params as the original, plus the Throwable.
    private PaymentResult paymentFallback(BigDecimal amount, Throwable ex) {
        return PaymentResult.queuedForRetry();  // degrade gracefully, don't hard-fail
    }
}
# application.yml -- the numbers that define "unhealthy"
resilience4j:
  circuitbreaker:
    instances:
      paymentGateway:
        sliding-window-size: 20            # judge on the last 20 calls
        failure-rate-threshold: 50         # open at >= 50% failures
        wait-duration-in-open-state: 60s   # cool-down before trying again
        permitted-number-of-calls-in-half-open-state: 3
⚠️
PitfallA fallback that silently returns fake data can be worse than an error — imagine a fallback returning "0 items in stock" and the UI showing everything sold out. Design fallbacks to degrade honestly: queue the work, show cached data marked as possibly stale, or tell the user to retry.

Retry with Backoff

Transient failures deserve another try, not an instant error.

In distributed systems, many failures are transient: a dropped packet, a node mid-restart, a one-second network blip. Failing the whole user operation over something that would succeed 200ms later is wasteful — so you retry. But retrying immediately is dangerous: if a service hiccups and a thousand callers all instantly retry, they hit it again at the exact same moment, in a synchronized wave that can knock it right back down (a retry storm). Exponential backoff spaces attempts out — 1s, 2s, 4s — and adding jitter (randomness) de-synchronizes the callers. Two rules: only retry errors that might actually resolve (timeouts yes, "invalid card number" no), and only retry operations that are safe to run twice (idempotent).
WhatsApp — delivery to a phone on a flaky network

A recipient is on a train drifting between cell towers. The first delivery attempt hits a dead zone and fails with a connection error. Marking the message "failed" over that would be absurd — the phone will be reachable again in seconds.

So delivery retries: wait 1 second, try again; still failing, wait 2; then 4. Three attempts covers the vast majority of brief network blips. Only when all attempts are exhausted does the system change strategy — the @Recover method hands the message to the offline-delivery queue, where it waits for the phone's next reconnect. Notice the escalation: retry handles the blip, the recover path handles the genuinely offline case, and the sender never sees either.

@Service
public class MessageDeliveryService {

    @Retryable(
        retryFor = ConnectException.class,     // ONLY transient errors -- never retry
        maxAttempts = 3,                        // business errors like "user blocked you"
        backoff = @Backoff(delay = 1000, multiplier = 2)   // 1s -> 2s -> 4s
    )
    public void deliver(Message message, String deviceId) {
        deviceClient.push(deviceId, message);
    }

    // Runs only after ALL 3 attempts have failed -- the escalation path.
    @Recover
    public void onDeliveryFailed(ConnectException ex, Message message, String deviceId) {
        offlineQueue.enqueue(message, deviceId);   // deliver when the phone reconnects
    }
}
// Requires @EnableRetry on a configuration class.
⚠️
PitfallRetrying a non-idempotent operation is how customers get charged twice: the charge succeeds but the response is lost, the client retries, and a second charge goes through. Fix it with an idempotency key — the client sends a unique ID per logical operation, and the server ignores duplicates of an ID it has already processed.

Rate Limiter

Cap how often a piece of code can run in a given window.

A rate limiter enforces a budget: at most N calls per time window, with excess calls rejected immediately (typically HTTP 429 "Too Many Requests") or briefly queued. It protects in two directions. Inbound: your public API caps each client so one buggy integration or scraper can't consume capacity meant for everyone. Outbound: when a partner API allows you 100 calls/minute, limiting yourself prevents them from banning you. Note the contrast with the circuit breaker: a breaker reacts to failures, a rate limiter enforces a budget on volume even when everything is perfectly healthy.
Instagram — public API per-developer limit

Thousands of third-party apps — schedulers, analytics dashboards, print services — call Instagram's public API. One developer ships a bug: a polling loop with no delay, hammering the profile endpoint hundreds of times per second. Without limits, that single API key degrades the platform for every other integration and for Instagram's own apps.

With a 200-calls-per-minute limit per key, the buggy app burns its own budget in the first second and gets clean 429 responses (with a "retry after" header) for the rest of the minute. Every other developer's app is completely unaffected — the blast radius of the bug is contained to the developer who wrote it.

@RestController
@RequestMapping("/api/public")
public class PublicApiController {

    @RateLimiter(name = "publicApi", fallbackMethod = "tooManyRequests")
    @GetMapping("/profile/{username}")
    public ResponseEntity<ProfileResponse> getProfile(@PathVariable String username) {
        return ResponseEntity.ok(profileService.getPublicProfile(username));
    }

    private ResponseEntity<ProfileResponse> tooManyRequests(String username, Throwable ex) {
        return ResponseEntity.status(429)
                .header("Retry-After", "60")     // tell well-behaved clients when to come back
                .build();
    }
}

        timeout-duration: 0          # reject instantly instead of queueing

Bulkhead & Timeout

Isolate failures so one slow dependency can't sink the whole app.

The name comes from shipbuilding: bulkheads divide a hull into watertight compartments so one breach floods a section, not the ship. In a service, the "hull" is your finite pool of request threads, and the flooding is slowness — a slow dependency causes calls to it to occupy threads for seconds instead of milliseconds, and without isolation those stuck threads are stolen from everything else the service does. A bulkhead caps how many concurrent calls one dependency may hold; a timeout caps how long any single call may take. Together they put a hard ceiling on the damage: at most N threads, for at most T seconds each.
Amazon — recommendations slow down, checkout must not

The product page calls the recommendations service for "customers also bought." Recommendations is nice-to-have; checkout is the business. One afternoon recommendations degrades — responses go from 50ms to 8 seconds. Do the math without protection: 200 request threads, each product-page render holding one for 8 seconds, and within a minute nearly the whole pool is stuck waiting on shelf-decoration while checkout requests queue behind them. Revenue stops because a widget got slow.

With a bulkhead of 10 and a 2-second timeout: at most 10 threads can ever be waiting on recommendations, and each gives up after 2 seconds. The other 190 threads keep serving checkout and search untouched. The product page quietly renders without the recommendations strip — a degradation most customers never even notice.

@Service
public class RecommendationClient {

    @Bulkhead(name = "recommendations")      // cap concurrent calls to this dependency
    @TimeLimiter(name = "recommendations")   // cap the duration of each call
    public CompletableFuture<List<Product>> getRecommendations(String userId) {
        return CompletableFuture.supplyAsync(() -> recommendationApi.fetch(userId));
    }
}
resilience4j:
  bulkhead:
    instances:
      recommendations:
        max-concurrent-calls: 10     # worst case: 10 threads tied up, never more
  timelimiter:
    instances:
      recommendations:
        timeout-duration: 2s         # worst case: tied up for 2s, never more
💡
TipThese patterns compose — production clients often stack @Bulkhead + @TimeLimiter + @CircuitBreaker + @Retry on one method. The layering: timeout bounds each call, bulkhead bounds concurrency, retry handles blips, and the breaker steps back when failures persist. Size timeouts from the dependency's real p99 latency plus headroom, not a guess.

Distributed Transactions — Saga Pattern

@Transactional only works within one database — sagas coordinate across services.

Chapter 12's @Transactional rests on one database's ability to roll back everything it did. Split order, inventory, and payment into separate microservices with separate databases and that guarantee evaporates — no single transaction can span them. A saga replaces the rollback with explicit business logic: a sequence of local transactions, where every step has a defined compensating action that undoes it (reserve stock ↔ release stock, charge card ↔ refund). If step 3 fails, the saga runs the compensations for steps 2 and 1 in reverse. Two coordination styles exist: orchestration (one service directs the sequence — easier to follow and debug) and choreography (services react to each other's events — more decoupled, harder to trace).
Amazon — an order across three services

Order-service asks inventory-service to reserve the last PlayStation, then asks payment-service to charge ₹49,990 — and the card is declined. The reservation already committed in inventory's own database; no shared transaction exists to undo it. Without compensation, that console is now locked "reserved" forever: unsellable, attached to no successful order — inventory silently leaking with every failed payment.

The saga makes the undo explicit: on payment failure, the orchestrator calls releaseStock, inventory returns to sellable, and the order is marked failed. The key mental shift — embrace that the system is briefly inconsistent (a few hundred milliseconds where stock is reserved but payment is undecided) and guarantee it always converges to a consistent state, forward or backward.

@Service
public class OrderSagaOrchestrator {

    public Order placeOrder(OrderRequest request) {
        // Step 1: local transaction inside inventory-service
        inventoryClient.reserveStock(request.sku(), request.qty());

        try {
            // Step 2: local transaction inside payment-service
            paymentClient.charge(request.orderId(), request.amount());
        } catch (PaymentFailedException ex) {
            // COMPENSATE step 1 -- the explicit "undo" that replaces rollback
            inventoryClient.releaseStock(request.sku(), request.qty());
            throw ex;
        }

        // Step 3: commit the order locally
        return orderRepository.save(Order.confirmed(request));
    }
}
⚠️
PitfallWhat if the orchestrator crashes between charging and compensating? Production sagas persist their progress (commonly via the outbox pattern — writing state and outgoing messages in one local transaction) so a restarted instance can resume or compensate. Compensations must also be idempotent: releasing the same reservation twice must be harmless.

Kafka Integration

A durable, ordered log that decouples producers from consumers.

Direct service-to-service calls create tight coupling: the caller must know who to call, wait for them, and handle their downtime. Kafka breaks that by putting a durable log in the middle. Producers append messages to a topic and move on — they neither know nor care who reads it. Consumers read at their own pace, and because messages persist on disk (for days, by your retention policy), a consumer that crashes resumes exactly where it left off instead of losing anything. Topics are split into partitions for parallelism, and messages with the same key always land in the same partition — which guarantees their order relative to each other. Separate consumer groups each get every message independently, which is what makes fan-out effortless.
WhatsApp — one sent message, three consumers

When you hit send, at least three things must happen: the message reaches the recipient's device, a push notification fires if their app is backgrounded, and the message lands in chat backup. Chaining these as direct calls would make sending as slow as the slowest step — and if the backup service is down, does sending fail entirely?

With Kafka, send-service publishes once to chat-messages, keyed by chat ID so each conversation's messages stay in order, and returns instantly. Three consumer groups — delivery, notifications, backup — each read the same message independently. When backup goes down for 20 minutes, deliveries and notifications continue untouched; backup restarts, finds its saved position in the log, and works through the backlog. Nothing lost, nobody blocked, no coordination between the three teams that own those consumers.

@Service
public class MessagePublisher {

    private final KafkaTemplate<String, ChatMessage> kafkaTemplate;

    public MessagePublisher(KafkaTemplate<String, ChatMessage> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publish(ChatMessage message) {
        // Key = chatId: all messages of one chat share a partition,
        // so their ORDER within that chat is guaranteed.
        kafkaTemplate.send("chat-messages", message.getChatId(), message);
        // returns immediately -- no waiting on any consumer
    }
}

// Different groupIds = independent consumers. Each group receives EVERY
// message and tracks its own reading position (offset) in the log.
@Component
public class DeliveryConsumer {
    @KafkaListener(topics = "chat-messages", groupId = "delivery-service")
    public void onMessage(ChatMessage message) {
        deviceClient.push(message);
    }
}

@Component
public class BackupConsumer {
    @KafkaListener(topics = "chat-messages", groupId = "backup-service")
    public void onMessage(ChatMessage message) {
        backupStore.append(message);   // down for 20 min? resumes from its offset
    }
}
⚠️
PitfallKafka's default guarantee is at-least-once: after a consumer crash mid-batch, some messages get redelivered. Design consumers to be idempotent — e.g. keying the backup store by message ID so a duplicate append is a harmless overwrite — rather than assuming exactly-once out of the box.

RabbitMQ Integration

Queue-based messaging with flexible routing rules.

RabbitMQ's model differs from Kafka's in a way that matters for choosing between them. Producers publish to an exchange with a routing key; the exchange routes each message into one or more queues based on binding rules; and each message in a queue is delivered to exactly one of that queue's workers, then acknowledged and gone. That "each message consumed once, by one worker" semantic makes RabbitMQ a natural work queue: pile up jobs, let a pool of workers share them, scale workers up when the queue grows. The rough rule of thumb: RabbitMQ for distributing jobs among workers with rich routing; Kafka for a replayable event log that many independent systems read.
Amazon — order confirmation emails

Checkout must respond in milliseconds; sending a templated confirmation email takes a couple of seconds and involves a third-party email provider that occasionally has a bad day. Doing it inline would make every customer wait on the slowest part — and couple checkout availability to the email provider's uptime.

So checkout publishes order.created to the order exchange and responds immediately. The message routes into the email queue, where a pool of five workers consumes it — each order's email handled by exactly one worker, never duplicated. During Diwali sales the queue deepens, the team scales to twenty workers, and the backlog drains; if the email provider goes down entirely, messages simply wait in the queue and every email still goes out once it recovers. Failed messages can be retried or shunted to a dead-letter queue for inspection.

@Service
public class OrderService {

    private final RabbitTemplate rabbitTemplate;

    public OrderService(RabbitTemplate rabbitTemplate) {
        this.rabbitTemplate = rabbitTemplate;
    }

    public void placeOrder(Order order) {
        orderRepository.save(order);
        // exchange + routing key -- bindings decide which queues receive it.
        // "order.#" bindings could also feed an analytics queue, a fraud queue...
        rabbitTemplate.convertAndSend("order-exchange", "order.created", order);
        // checkout responds to the customer NOW; email happens on its own time
    }
}

@Component
public class OrderEmailWorker {
    // Five instances of this worker share the queue -- RabbitMQ delivers each
    // message to exactly one of them. Scaling = just add more workers.
    @RabbitListener(queues = "order-email-queue", concurrency = "5")
    public void onOrderCreated(Order order) {
        emailService.sendConfirmation(order);   // ack on success; requeue/DLQ on failure
    }
}
💡
TipAlways configure a dead-letter queue: a message that keeps failing (say, a malformed address crashing the worker) would otherwise be redelivered forever, jamming the queue. After N failed attempts it moves to the DLQ, where the team can inspect and replay it — one poison message never blocks the other ten thousand.

WebSocket / STOMP

A persistent, two-way connection for instant updates.

Plain HTTP is one-directional: the client asks, the server answers, the connection closes. For live features that's backwards — the server knows when something happened, but has no way to tell anyone. Polling ("anything new?" every 2 seconds) fakes it, at the cost of constant wasted requests and up to 2 seconds of built-in lag. A WebSocket upgrades one HTTP request into a persistent, two-way channel: either side can push data the instant it exists. STOMP layers a tiny pub/sub protocol on top — clients subscribe to topics like /topic/chat/42, and the server broadcasts to every subscriber — so you're not inventing your own message format over a raw socket.
WhatsApp — the typing indicator

"Rahul is typing…" is only worth showing if it appears within a fraction of a second of Rahul actually typing. Polling can't get there: even at an aggressive 1-second interval you'd average half a second of lag while generating a relentless stream of mostly-empty responses from millions of idle chats — the worst of both worlds.

Over WebSockets, each open chat holds a live connection. Rahul's phone sends one tiny frame to /app/typing/chat42; the server broadcasts it to /topic/typing/chat42; everyone viewing that chat renders "typing…" within milliseconds. Total network cost: one frame in, one frame out per subscriber — only when someone actually types. The same channel carries message delivery, read receipts (blue ticks), and presence updates.

@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws-chat").withSockJS();  // clients connect here once
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic");               // server -> subscribers
        registry.setApplicationDestinationPrefixes("/app");  // client -> @MessageMapping
    }
}

@Controller
public class TypingController {

    // Client SENDS to /app/typing/chat42; return value is BROADCAST to
    // every client subscribed to /topic/typing/chat42. That's the whole flow.
    @MessageMapping("/typing/{chatId}")
    @SendTo("/topic/typing/{chatId}")
    public TypingEvent onTyping(@DestinationVariable String chatId, TypingEvent event) {
        return event;
    }
}
💡
TipThe built-in simple broker lives in one JVM — with multiple app instances, a user connected to instance A never hears broadcasts originating on instance B. In production, back STOMP with an external broker (RabbitMQ's STOMP plugin, or Redis pub/sub) so every instance shares the same topics.

Service Discovery — Eureka

Services find each other by name, not hard-coded IPs.

In a container world, IP addresses are disposable: instances scale from 3 to 30 during a sale, crash and get replaced, and every replacement gets a fresh address. Hard-coding "inventory-service lives at 10.0.4.17" breaks by lunchtime. A service registry fixes this with three moves: each instance registers itself by logical name on startup ("I'm inventory-service at this address"), sends heartbeats so dead instances fall out of the registry automatically, and callers look up a current healthy instance by name at call time — with client-side load balancing spreading requests across whatever instances exist right now. Eureka (from Netflix) is the classic Spring Cloud implementation; Kubernetes provides the same idea natively through Services and DNS.
Amazon — checkout finds inventory during a sale

At 9 AM, inventory-service runs 3 instances. The Great Indian Festival sale starts at noon; autoscaling pushes it to 30 instances, each on an address that didn't exist this morning. By midnight it's back to 3, and overnight one of those crashes and is replaced. Any hard-coded address list is wrong within hours — and updating config on every caller for every scaling event is madness.

With Eureka, none of that churn is checkout-service's problem. Every inventory instance registers on boot and heartbeats every 30 seconds; instances that die simply stop heartbeating and are evicted. Checkout asks for "inventory-service" by name and gets load-balanced across whatever healthy instances exist at that instant — 3 at breakfast, 30 at noon, without a single configuration change anywhere.

// On each inventory-service instance: register with Eureka at startup.
@SpringBootApplication
@EnableDiscoveryClient
public class InventoryServiceApplication { }

// On the caller: @LoadBalanced teaches the client to resolve logical names
// via Eureka and spread calls across healthy instances (round-robin).
@Configuration
public class HttpClientConfig {
    @Bean
    @LoadBalanced
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

@Service
public class CheckoutClient {
    private final RestTemplate restTemplate;

    public CheckoutClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public boolean reserveStock(String sku, int qty) {
        // "inventory-service" is a LOGICAL name, not a host --
        // resolved to a live instance at the moment of each call.
        return restTemplate.postForObject(
            "http://inventory-service/api/reserve",
            new ReserveRequest(sku, qty), Boolean.class);
    }
}
💡
TipInterview nuance: on Kubernetes you usually don't run Eureka — the platform's built-in Service + DNS discovery does the same job (http://inventory-service just resolves). Eureka matters when you're off Kubernetes, or need client-side load-balancing decisions the platform doesn't make for you.

API Gateway — Spring Cloud Gateway

One front door that routes to every microservice behind it.

Exposing 40 microservices directly to the internet creates two problems: clients must know 40 addresses, and every cross-cutting concern — authentication, rate limiting, TLS, logging — must be implemented 40 times, identically, forever. A gateway is the single entry point that fixes both: clients see one host, the gateway matches each request against route predicates (path, header, method) and forwards to the right internal service, applying shared filters — validate the JWT, enforce rate limits, strip prefixes, log — exactly once, in one place. Internal services then live on a private network, unreachable except through the gateway.
Amazon — one host, forty services

The mobile app talks only to api.amazon.com. Behind that host, /products/** lands on product-service, /orders/** on order-service, /cart/** on cart-service — the app has no idea these are separate systems, which means the backend team can split, merge, or rewrite services freely without ever shipping an app update.

The gateway is also where platform-wide policy lives: every request's JWT is validated once at the door (services behind it trust the gateway's verdict), per-customer rate limits are enforced where all traffic converges, and when the team migrates reviews out of product-service into a new reviews-service, it's a one-line route change at the gateway — invisible to every client in the world.

spring:
  cloud:
    gateway:
      routes:
        - id: products-route
          uri: lb://product-service        # lb:// = resolve via discovery + load-balance
          predicates:
            - Path=/products/**
        - id: orders-route
          uri: lb://order-service
          predicates:
            - Path=/orders/**
          filters:
            - StripPrefix=1                 # /orders/123 -> /123 inside the service
      default-filters:
        - name: RequestRateLimiter          # enforced once, at the front door,
          args:                             # for every route behind the gateway
            redis-rate-limiter.replenishRate: 100
            redis-rate-limiter.burstCapacity: 200
⚠️
PitfallThe gateway is on every request's path — it must be the most boring, stateless, horizontally-scaled thing you run. Teams that let business logic creep into gateway filters ("just this one order-enrichment step...") end up with a fragile god-service that every deploy risks. Route, authenticate, limit, log — and keep business logic in the services.

Centralized Config — Spring Cloud Config

One place to change a setting across every running instance.

Chapter 8's profiles solve per-environment config for one service — but with 40 services each carrying their own application.yml, shared settings drift and every change means a rebuild-and-redeploy. Spring Cloud Config centralizes it: a Config Server serves configuration out of a Git repository, each service fetches its own settings (by app name + profile) at startup, and beans marked @RefreshScope can pick up changes live — no restart. Git as the backing store is the quiet superpower: every config change is a commit, with an author, a diff, a review, and an instant git revert when a change goes wrong.
Instagram — feature flag on the new feed ranking

The team ships a new feed-ranking algorithm dark — deployed but disabled behind a flag. Enabling it for a small slice of users must be instant, and disabling it must be even faster if engagement metrics dip. Redeploying 200 feed-service instances to flip a boolean would take twenty minutes each way — an eternity when the feed is misbehaving in production.

With central config, the flag lives in one Git file. Enabling: commit rollout-percent: 5, trigger a refresh, and all 200 instances flip within seconds. When metrics look wrong at 2 AM, rollback is git revert + refresh — feed back to the old algorithm in under a minute, full audit trail of who changed what and when included for free.

# In the CONFIG GIT REPO: feed-service.yml -- one file governs all instances
feed:

    enabled: true
    rollout-percent: 5

# In feed-service itself: just one line pointing at the Config Server
# application.yml:
#   spring.config.import: configserver:http://config-server:8888

@Component
public class FeedRankingConfig {

    private int rolloutPercent;

Declarative REST Clients — OpenFeign

Call another service like a local method — no manual HTTP plumbing.

Every hand-rolled RestTemplate call repeats the same ceremony: build the URL, set headers, serialize the body, deserialize the response, translate errors. Feign inverts it — you write only an interface describing the remote API in the same annotations you'd use on a controller (@PostMapping, @RequestBody), and Feign generates the entire HTTP client implementation at startup. Point it at a Eureka logical name and load balancing comes along free; wire in Resilience4j and the circuit breakers from Chapter 27 wrap every call. The calling code reads like it's calling a local service — which is exactly the point.
Amazon — checkout's clients, before and after

Checkout-service calls inventory, payment, and shipping. As RestTemplate code, that's three families of URL-building, header-setting, error-mapping boilerplate — subtly inconsistent with each other, each drifting as the remote APIs evolve, none of it business logic.

As Feign clients, each remote API is one small interface that reads like documentation: method, path, request, response. CheckoutService injects them like any local bean and calls inventoryClient.reserveStock(...) — the fact that a network hop, service discovery lookup, load-balancing decision, and JSON round-trip happen behind that line is entirely Feign's business.

// The ENTIRE client. No implementation class exists anywhere -- Feign
// generates it at startup from these annotations.
@FeignClient(name = "inventory-service")   // logical name -> Eureka + load balancing
public interface InventoryClient {

    @PostMapping("/api/reserve")
    boolean reserveStock(@RequestBody ReserveRequest request);

    @GetMapping("/api/stock/{sku}")
    StockLevel getStock(@PathVariable("sku") String sku);
}

@Service
public class CheckoutService {

    private final InventoryClient inventoryClient;   // injected like any local bean

    public CheckoutService(InventoryClient inventoryClient) {
        this.inventoryClient = inventoryClient;
    }

    public void checkout(String sku, int qty) {
        // Reads like a local call; is actually discovery + LB + HTTP + JSON.
        if (!inventoryClient.reserveStock(new ReserveRequest(sku, qty))) {
            throw new OutOfStockException(sku);
        }
    }
}
// Requires @EnableFeignClients on a configuration class.

Distributed Tracing

Follow one request as it hops across every microservice it touches.

In a monolith, one stack trace tells the whole story of a slow request. Split the work across six services and that story shatters into six unconnected log files. Tracing reconnects it: when a request first enters the system, it's assigned a trace ID, and that ID rides along in a header through every HTTP call and Kafka message the request triggers. Each unit of work (a controller handling, a client call, a DB query) records a span — its name, duration, and parent — and a backend like Zipkin or Jaeger assembles all spans sharing a trace ID into one waterfall diagram of the request's entire journey. Micrometer Tracing does the propagation automatically for Spring's HTTP clients, Kafka, and logging; you mostly just add the dependency.
YouTube — "uploads are slow," but where?

Users report uploads taking 6 seconds. The request path: gateway → upload-service → transcoding-service → storage-service, plus a metadata write. Without tracing, diagnosis is archaeology — four teams grepping four log systems, trying to match timestamps, each concluding "our part looks fine."

With tracing, an engineer pulls up one slow trace in Zipkin and reads the waterfall: gateway 3ms, upload-service 40ms, transcoding submit 25ms... and there it is — storage-service's span shows 5.4 seconds, and its child span reveals why: an internal retry loop hitting a misconfigured bucket region three times before succeeding. Diagnosis in minutes, the right team paged, and the trace ID printed in every log line means the matching logs across all four services are one search away.

// build.gradle -- this pair is most of the work:
// implementation 'io.micrometer:micrometer-tracing-bridge-brave'   // creates/propagates traces
// implementation 'io.zipkin.reporter2:zipkin-reporter-brave'       // ships them to Zipkin

@RestController
public class UploadController {

    private static final Logger log = LoggerFactory.getLogger(UploadController.class);

    @PostMapping("/api/videos/upload")
    public UploadResponse upload(@RequestBody UploadRequest request) {
        // This log line automatically includes [app,traceId,spanId] --
        // the SAME traceId appears in transcoding-service's and
        // storage-service's logs for this request. Correlation for free.
        log.info("starting upload for {}", request.title());

        return transcodingClient.submit(request);
        // The outgoing call carries the trace ID in headers automatically --
        // no manual propagation code anywhere.
    }
}
💡
TipProduction systems sample — tracing every request at high volume is costly, so typically ~10% of traces are kept (plus, ideally, everything that errored). Set it via management.tracing.sampling.probability: 0.1. For debugging a specific issue, you can temporarily raise it.

Metrics & Dashboards

Turn app behavior into numbers a dashboard can graph and alert on.

Tracing explains one request; metrics summarize all of them — request rates, error percentages, latency percentiles, queue depths — as cheap numeric time series you can graph and, crucially, alert on. The pipeline: Micrometer is the metrics API inside your app (think "SLF4J, but for metrics"), Actuator exposes everything at /actuator/prometheus, Prometheus scrapes that endpoint every few seconds and stores the history, Grafana graphs it, and Alertmanager pages a human when a threshold breaks. Spring Boot auto-instruments a lot (every HTTP endpoint's rate and latency, JVM memory, connection pools) — the skill is adding the business metrics that generic instrumentation can't know about.
Amazon — catching a silent checkout failure

A bad deploy at 14:00 makes one payment provider integration start failing — but only for that provider, roughly 8% of checkouts. No exception storm, servers healthy, HTTP dashboards green: the failures return well-formed error responses. Generic infrastructure metrics literally cannot see this problem.

The business metric can. checkout.failures, tagged by failure reason and provider, jumps from ~2/minute to 40/minute at 14:01. The Grafana alert fires by 14:03, the on-call engineer sees the spike is isolated to one provider tag, correlates it with the 14:00 deploy, and rolls back — total customer impact, a few minutes. Without that one counter, the first signal would have been angry customers hours later.

@Service
public class CheckoutService {

    private final MeterRegistry registry;

    public CheckoutService(MeterRegistry registry) {
        this.registry = registry;
    }

    public void checkout(Order order) {
        // Timer: counts calls AND tracks latency percentiles (p95, p99)
        Timer.Sample sample = Timer.start(registry);
        try {
            processCheckout(order);
        } catch (PaymentException ex) {
            // Tags turn one counter into a sliceable dashboard dimension:
            // failures BY reason, BY provider -- that's what located the bug.
            registry.counter("checkout.failures",
                    "reason", ex.getReason(),
                    "provider", order.getPaymentProvider()).increment();
            throw ex;
        } finally {
            sample.stop(registry.timer("checkout.duration"));
        }
    }
}
⚠️
PitfallNever use an unbounded value (user ID, order ID) as a tag — each distinct tag value creates a separate time series, and a million users means a million series: a "cardinality explosion" that takes down Prometheus itself. Tag with small, fixed sets: reason, provider, region, status.

OAuth2 / JWT Authentication

Stateless login: the token itself proves who you are.

Session-based auth stores login state on the server and hands the browser a cookie pointing at it — fine for one server, awkward for twenty microservices that would all need access to a shared session store. A JWT (JSON Web Token) flips the model: after login, the auth server issues a token containing the user's identity and claims (ID, roles, expiry), cryptographically signed. The client sends it as a bearer header on every request, and any service can verify the signature with the auth server's public key and trust the claims — no session store, no call back to the auth server, no shared state. That self-contained verification is what makes JWTs the default currency of microservice authentication. OAuth2 is the surrounding protocol standardizing how tokens are issued and refreshed (and how "Log in with Google" works).
Instagram — one login, twenty services

You log in once on your phone. Over the next hour, the app talks to the feed service, profile service, DM service, notification service — separate deployments, separate teams, separate databases. With sessions, every one of them would need to check a shared session store on every request: a bottleneck and a single point of failure touched by all traffic.

With JWT, login returns a signed token asserting "user 8271, expires in 15 minutes." Each service verifies the signature locally — a fast cryptographic check, no network call — and proceeds. The DM service never asks the auth service "is this session valid?"; the token's signature already answers it. A refresh token quietly obtains new access tokens so the user isn't re-prompted every 15 minutes.

// AUTH SERVICE: verify credentials, issue the signed token.
@RestController
public class AuthController {

    @PostMapping("/api/auth/login")
    public TokenResponse login(@RequestBody LoginRequest request) {
        User user = authService.authenticate(request);   // check password
        String jwt = jwtService.generateToken(user.getId(), user.getRoles(),
                                              Duration.ofMinutes(15));
        return new TokenResponse(jwt);
    }
}

// EVERY OTHER SERVICE: three lines of config = "verify JWTs, trust their claims."
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .anyRequest().authenticated())
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));
        return http.build();
        // Signature verified locally against the auth server's public key
        // (fetched once from its JWKS endpoint). No per-request network call.
    }
}
⚠️
PitfallA JWT can't be un-issued — it stays valid until expiry even if the user logs out or is banned. That's why access tokens are kept short-lived (minutes), with refresh tokens (which can be revoked server-side) obtaining new ones. Long-lived access tokens turn one leaked header into a long-term compromise.

CORS Configuration

Tell browsers exactly which other origins may call your API.

Browsers enforce the same-origin policy: JavaScript on site-a.com can't read responses from api.site-b.com unless site-b explicitly allows it. This isn't the browser being difficult — it's what stops a malicious page you accidentally opened from silently calling your bank's API with your logged-in cookies. CORS (Cross-Origin Resource Sharing) is the server's allowlist: response headers declaring which origins, methods, and headers are permitted. For non-trivial requests the browser first sends a preflight — an OPTIONS request asking permission — and only proceeds if the server's CORS headers approve. The critical mental model: the error appears in the frontend console, but the fix is always on the backend.
Amazon — seller dashboard blocked by the browser

The seller dashboard is a React app on seller.amazon.com; the product API lives on api.amazon.com — different subdomains, therefore different origins. The frontend team deploys, opens the dashboard, and every API call fails with the console error every developer eventually meets: "blocked by CORS policy: No 'Access-Control-Allow-Origin' header." Confusingly, the same API works perfectly from Postman — because Postman isn't a browser and enforces no same-origin policy.

The fix is one backend config class: allow origin https://seller.amazon.com, the needed methods, and the Authorization header (the dashboard sends JWTs from Chapter 41). Preflights now succeed, and — importantly — a malicious evil-shop.com still gets blocked, because it's not on the allowlist. CORS didn't get disabled; it got configured.

@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://seller.amazon.com")   // exact allowlist -- not "*"
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("Authorization", "Content-Type")
                .maxAge(3600);   // cache the preflight verdict for an hour
    }
}
⚠️
PitfallallowedOrigins("*") is the tempting "make the error go away" fix — and it invites any website on the internet to call your API from its users' browsers. Worse, wildcard + credentials is forbidden by the spec entirely. List your real origins explicitly; in Spring Security setups, also ensure CORS is enabled in the filter chain or preflights get rejected before your config is consulted.
Part 13 — Batch & Files

Spring Batch

Process millions of records reliably, with restart-from-failure built in.

A naive "loop over all rows" job has three famous failure modes: it loads too much into memory, a crash at record 4.7 million loses all progress, and rerunning it double-processes everything before the crash. Spring Batch is the framework answer. Work is structured as a Job of Steps, each step a read → process → write loop over chunks — read 1,000 items, process them, write them, commit, repeat — so memory stays flat and progress commits incrementally. The JobRepository persists execution state in database tables: which job ran, with which parameters, how far each step got. That's the superpower — a failed job restarts from the last committed chunk, not from zero.
Amazon — nightly seller payouts, crash included

Every night, a job aggregates the day's millions of order rows into payout records for hundreds of thousands of sellers — money, so correctness is non-negotiable: no seller skipped, none paid twice. At 2:47 AM, four million records in, the database fails over and the job dies.

With Spring Batch, the night shift's response is one command: restart the job. The JobRepository knows exactly which chunks committed, the step resumes from chunk 4,000, and the already-written payouts are neither redone nor duplicated. The 8 AM payout deadline is met. A hand-rolled loop would have meant a very different morning: forensic SQL to figure out who was already paid, and a hand-edited rerun script — with real money riding on getting it right.

@Bean
public Job payoutReportJob(JobRepository jobRepository, Step calculatePayoutsStep) {
    return new JobBuilder("payoutReportJob", jobRepository)
            .start(calculatePayoutsStep)      // real jobs chain multiple steps
            .build();
}

@Bean
public Step calculatePayoutsStep(JobRepository jobRepository,
                                  PlatformTransactionManager tx,
                                  ItemReader<Order> orderReader,          // pages from the DB
                                  ItemProcessor<Order, Payout> processor, // order -> payout
                                  ItemWriter<Payout> payoutWriter) {      // bulk insert
    return new StepBuilder("calculatePayoutsStep", jobRepository)
            .<Order, Payout>chunk(1000, tx)   // read/process/write, commit every 1000
            .reader(orderReader)               // <- restart resumes from the last
            .processor(processor)              //    committed chunk, automatically
            .writer(payoutWriter)
            .build();
}
💡
TipJob parameters (like the run date) identify a job instance — rerunning with the same parameters means "resume that instance," a new date means "new run." Trigger nightly execution with Chapter 15's @Scheduled, passing today's date as the parameter: scheduling starts the job, Batch guarantees its integrity.

File Upload Handling

Accepting large binary files without loading them entirely into memory.

HTTP sends files as multipart/form-data — the request body carries one or more file "parts" alongside regular form fields. Spring's MultipartFile hands each part to your controller, and its most important method is getInputStream(): it lets you stream the bytes onward (to disk, to S3) without ever materializing the whole file in heap memory. The other essential is centrally-configured size limits, enforced by the framework before your code runs — the difference between "a too-large upload gets a clean 413 response" and "a too-large upload takes down the JVM."
YouTube — a 4 GB video meets a 2 GB heap

A creator uploads a 4 GB video to a service instance running with a 2 GB heap. If the handler naively calls file.getBytes() — the whole file as one byte array — the math doesn't work: OutOfMemoryError, instance dead, and every other creator's in-flight upload on that instance dies with it. Two concurrent large uploads would kill it even faster.

The streaming version never holds more than a small buffer: bytes flow from the network socket through the input stream into object storage in chunks, heap usage flat regardless of file size — ten simultaneous 4 GB uploads are fine. The 5 GB configured cap rejects oversized requests up front with a clean error, and the controller returns quickly by kicking off transcoding asynchronously (Chapter 14's @Async) instead of making the creator's browser wait on processing.

@RestController
@RequestMapping("/api/videos")
public class VideoUploadController {

    private final StorageService storageService;
    private final TranscodingService transcodingService;

    public VideoUploadController(StorageService storageService,
                                  TranscodingService transcodingService) {
        this.storageService = storageService;
        this.transcodingService = transcodingService;
    }

    @PostMapping("/upload")
    public ResponseEntity<UploadResponse> upload(@RequestParam("file") MultipartFile file)
            throws IOException {
        if (file.isEmpty()) {
            return ResponseEntity.badRequest().build();
        }

        // STREAM to storage -- never file.getBytes() for large files.
        // Heap holds only a small buffer, whatever the file size.
        String storedPath = storageService.store(
                file.getInputStream(), file.getOriginalFilename(), file.getSize());

        transcodingService.transcodeAsync(storedPath);   // @Async -- don't block the upload

        return ResponseEntity.ok(new UploadResponse(storedPath));
    }
}
# Enforced by the framework BEFORE your controller runs --
# an oversized request gets a clean 413, not an OutOfMemoryError.

  servlet:
    multipart:
      max-file-size: 5GB
      max-request-size: 5GB
💡
TipAt real scale, big files often skip your server entirely: the API returns a pre-signed URL and the client uploads straight to S3/GCS, so your service handles metadata while the storage provider handles the bytes. Know both patterns — "when would you use pre-signed URLs?" is a favorite interview follow-up.

OpenAPI / Swagger

Interactive, always-up-to-date API documentation generated from your code.

Hand-written API docs share a destiny: they drift. Someone renames a field, adds a parameter, changes a status code — the wiki page doesn't hear about it, and consumers integrate against documentation that's quietly wrong. OpenAPI fixes the incentive problem by generating the spec from the code itself: springdoc scans your controllers, request/response types, and validation annotations at runtime and produces a machine-readable spec plus Swagger UI — a browsable page where every endpoint can be tried live, right from the browser. The spec being machine-readable is the second win: consumers can generate typed client SDKs from it instead of hand-writing HTTP calls against prose.
Amazon — the Marketplace API and ten thousand sellers

External sellers and tool vendors integrate against Amazon's Marketplace API. At that scale, documentation quality is support-ticket volume: every ambiguity — "is this field required?", "what does a 409 mean here?" — turns into thousands of identical questions. And a hand-maintained doc for hundreds of evolving endpoints would be permanently, somewhere, wrong.

Generated docs can't drift: when a developer adds a fulfillmentType parameter, the spec updates the moment the code deploys, because the code is the source. A seller's developer opens Swagger UI, reads the endpoint, tries a real call against sandbox with their token, sees the exact response shape — and then generates a typed Java or Python client from the spec instead of hand-rolling requests. The @Operation annotations layer human explanation on top of the auto-detected structure.

// build.gradle: implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.x'
// That single dependency = /v3/api-docs (the spec) + /swagger-ui.html (the UI).

@RestController
@RequestMapping("/api/products")
@Tag(name = "Products", description = "Marketplace product catalog operations")
public class ProductController {

    @Operation(
        summary = "Get a single product by ID",
        description = "Returns full product details. Requires the catalog:read scope."
    )
    @ApiResponse(responseCode = "200", description = "Product found")
    @ApiResponse(responseCode = "404", description = "No product with this ID")
    @GetMapping("/{id}")
    public ResponseEntity<ProductResponse> getProduct(@PathVariable String id) {
        return productService.findById(id)
                .map(p -> ResponseEntity.ok(ProductResponse.from(p)))
                .orElse(ResponseEntity.notFound().build());
    }
    // Validation annotations from Chapter 9 (@NotBlank, @Size) show up in the
    // generated schema automatically -- consumers see the real constraints.
}
⚠️
PitfallSwagger UI on a production, internet-facing service is a free map of your entire attack surface. Expose it freely in dev and staging; in production, disable it (springdoc.swagger-ui.enabled: false) or lock it behind authentication — publish the spec to partners through a proper developer portal instead.
